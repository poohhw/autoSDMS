# Git 커밋 & 푸시 자동화 (한글)

이 스킬은 다음 워크플로우를 자동으로 실행합니다:

1. 변경된 파일 감지
2. 변경사항 분석 및 한글 커밋 메시지 제안
3. 사용자 승인 후 커밋 실행
4. 커밋 로그 확인
5. 사용자 동의 후 `origin/main`으로 푸시

## 워크플로우

### Step 1: 변경사항 확인

먼저 현재 git 상태를 확인합니다:

```bash
git status
```

**변경사항이 없는 경우**: 스킬은 종료되고 "변경사항이 없습니다"라는 메시지를 출력합니다.

### Step 2: 변경사항 분석 및 커밋 메시지 제안

변경사항이 있으면 `git diff`를 실행하여 구체적인 변경내용을 확인합니다:

```bash
git diff
```

분석 결과를 바탕으로 **한글 자유형식 커밋 메시지**(한 줄 요약)를 제안합니다.

예:
- "로그인 기능 버그 수정"
- "사용자 프로필 페이지 UI 개선"
- "데이터베이스 연결 타임아웃 문제 해결"

### Step 3: 사용자 확인 및 메시지 조정

사용자에게 제안된 커밋 메시지를 제시하고, 다음 중 하나를 선택하도록 요청합니다:

- ✅ 제안된 메시지 그대로 사용
- ✏️ 메시지 수정 (사용자가 새 메시지 제공)

### Step 4: 커밋 실행

사용자 승인 후 커밋을 실행합니다:

```bash
git add .
git commit -m "사용자가 확인한 메시지"
```

**실패 처리**: Commit이 실패하면:
- ❌ 에러 로그 출력
- 💡 해결 방법 안내 제공 (예: 구성 설정 확인 등)
- 스킬 중단

### Step 5: 커밋 로그 확인

커밋 성공 후 최근 커밋 로그를 확인합니다:

```bash
git log -1 --oneline
```

사용자에게 커밋 정보를 보여줍니다.

### Step 6: Push 동의 구하기

사용자에게 "이제 `origin/main`으로 푸시하시겠습니까?"라고 확인합니다:

- ✅ 예 → push 실행
- ❌ 아니요 → 종료 (push 하지 않음)

### Step 7: Push 실행

사용자가 동의하면 `origin/main`으로 푸시합니다:

```bash
git push origin main
```

**Push 성공**: "✅ 푸시 완료"라는 메시지 출력

**Push 실패 (충돌)**: 
- ❌ 에러 로그 출력
- 💡 해결 방법 제시:
  - "리모트 변경사항을 먼저 가져오세요: `git pull origin main`"
  - "충돌이 있으면 해결 후 다시 커밋하세요"
  - "필요시 `git push origin main --force-with-lease`를 사용할 수 있습니다"

---

## Edge Cases 처리

| 상황 | 동작 |
|------|------|
| 변경사항 없음 | "변경사항이 없습니다. 스킬 종료" |
| Commit 실패 | 에러 로그 + 해결 방법 제시 후 중단 |
| Push 충돌 | 에러 로그 + 해결 방법 제시 후 중단 |
| 사용자 메시지 수정 요청 | 새 메시지로 커밋 실행 |
| Push 취소 | push하지 않고 스킬 종료 |

## 커밋 메시지 작성 가이드

제안 시 다음을 고려합니다:

- **자유형식**: 정형화된 prefix 불필요 (Conventional Commits 미적용)
- **한 줄 요약**: 간결하고 명확하게 (70자 이내 권장)
- **한글**: 모든 메시지는 한글로 작성
- **현재형**: "고쳤다" 대신 "고정" 또는 "버그 수정"

## 사용 예시

**사용자**: "변경사항을 커밋하고 푸시해줄 수 있어?"

**스킬 동작**:
1. 변경 파일 확인 ✓
2. 메시지 제안: "사용자 인증 로직 개선"
3. 사용자: "좋아, 그대로 커밋해"
4. 커밋 실행 ✓
5. 로그 확인: `abc1234 사용자 인증 로직 개선`
6. 푸시 동의 요청: "origin/main으로 푸시할까요?"
7. 사용자: "응, 해줘"
8. 푸시 실행 ✓
9. "✅ 완료!"

## 실행 지시 (당신의 작업)

다음 단계를 순서대로 진행합니다:

### 1단계: 변경사항 확인
- `git status` 실행
- 변경사항이 없으면 "변경사항이 없습니다"라고 알리고 종료
- 변경사항이 있으면 `git diff` 실행하여 변경내용 파악

### 2단계: 커밋 메시지 제안
- 변경사항을 분석하여 **한글 자유형식 한 줄 요약** 커밋 메시지 제안
- 변경된 파일들을 확인하고, 주요 변경사항에 기반하여 작성

### 3단계: 사용자 확인 (AskUserQuestion)
사용자에게 다음 질문을 제시:
```
질문: "커밋 메시지가 좋나요?"
옵션:
- "좋아, 그대로 커밋해" (제안 메시지 사용)
- "메시지 수정하고 싶어" (사용자가 새 메시지 입력)
```

### 4단계: 커밋 실행
사용자 선택에 따라:
- **옵션 1 선택**: `git add .` + `git commit -m "제안 메시지"` 실행
- **옵션 2 선택**: `git add .` + `git commit -m "사용자 입력 메시지"` 실행

커밋 실패 시:
```
❌ 에러 로그 출력
💡 일반적인 해결 방법:
   - git config user.name 설정 확인
   - git config user.email 설정 확인
   - 파일 권한 확인
```

### 5단계: 커밋 로그 확인
- `git log -1 --oneline` 실행하여 커밋 정보 표시
- "커밋이 완료되었습니다: [커밋 정보]"라고 알림

### 6단계: 푸시 동의 구하기 (AskUserQuestion)
사용자에게 다음 질문 제시:
```
질문: "이제 origin/main으로 푸시할까요?"
옵션:
- "예, 푸시해줘"
- "아니요, 나중에 할래"
```

### 7단계: 푸시 실행
사용자가 "예" 선택 시:
- `git push origin main` 실행

**Push 성공**:
```
✅ 푸시 완료!
```

**Push 실패 (충돌/거부)** 시:
```
❌ 에러 로그 출력
💡 해결 방법:
   1. git pull origin main 으로 리모트 변경사항 가져오기
   2. 충돌이 있으면 해결하고 다시 커밋
   3. git push origin main 재시도
   4. 필요시 git push origin main --force-with-lease 사용 가능
```

**사용자가 "아니요" 선택** 시:
```
푸시를 취소했습니다. 나중에 필요하면 언제든 "푸시해줄래?"라고 말씀해주세요.
```
